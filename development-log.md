## 开发策略说明

本项目采用“先落地，再完善”的开发策略。

原因：

- 课程实训项目需要尽早形成可运行、可演示的系统。
- 先做界面可以帮助团队统一页面结构、菜单、路由和交互流程。
- 页面骨架完成后，各成员可以围绕明确入口并行开发后端功能。
- 前期使用模拟数据能减少数据库和业务逻辑未完成带来的阻塞。
- 后期再逐步替换为真实数据库、真实文件上传下载和权限控制，风险更可控。

答辩表达：

> 我们没有一开始就直接堆复杂功能，而是先完成可访问的系统原型，确定整体页面和业务流程，再逐步把模拟数据替换为真实数据，把页面操作接入数据库、文件系统和权限控制。这样能保证项目每个阶段都有可运行成果，也方便团队协作和阶段性验收。

答辩可讲点：

### 阶段 4：分类和标签真实 CRUD

日期：2026-06-29

阶段目标：

- 将分类管理页面从模拟数据切换为数据库真实数据。
- 将标签管理页面从模拟数据切换为数据库真实数据。
- 完成分类和标签的新增、编辑、删除。
- 保持管理员权限控制，普通用户不能访问分类和标签管理页面。

完成内容：

- 新增 `CategoryController`，处理分类列表、新增、编辑、删除。
- 新增 `TagController`，处理标签列表、新增、编辑、删除。
- 改造 `HomeController`，将分类和标签数据从数据库读取，并计算关联文档数。
- 修改 `app.html`，在分类/标签管理区域加入创建表单、编辑表单、删除按钮、成功/失败提示。
- 修改 `app.css`，补充管理表单、行操作按钮、颜色圆点等样式。
- 分类删除前检查 `document.category_id` 是否仍被文档引用。
- 标签删除时先删除 `document_tag_rel` 关联记录，再删除标签本身。
- 继续复用阶段 2 的管理员权限拦截，普通用户访问 `/categories`、`/tags` 会被拒绝。

技术与方案：

- 后端采用 Spring MVC 表单提交方式，新增和更新使用 `POST` 请求。
- 分类路由：
  - `GET /categories`：分类列表。
  - `GET /categories?editId={id}`：进入编辑状态。
  - `POST /categories`：新增分类。
  - `POST /categories/{id}/update`：更新分类。
  - `POST /categories/{id}/delete`：删除分类。
- 标签路由：
  - `GET /tags`：标签列表。
  - `GET /tags?editId={id}`：进入编辑状态。
  - `POST /tags`：新增标签。
  - `POST /tags/{id}/update`：更新标签。
  - `POST /tags/{id}/delete`：删除标签。
- 数据访问继续使用 MyBatis-Plus：
  - `DocumentCategoryService` 操作 `document_category`。
  - `DocumentTagService` 操作 `document_tag`。
  - `DocumentService` 统计分类下文档数量。
  - `DocumentTagRelService` 统计和解除标签关联。
- 查询条件使用 `Wrappers.lambdaQuery()`，避免硬编码字段名，提高重构安全性。
- 表单提交后使用 `RedirectAttributes` 传递一次性成功/失败提示，避免刷新页面重复提交。

实现细节：

- 分类新增时会校验名称不能为空、名称不能重复。
- 分类编辑时会排除当前 ID 后检查名称重复。
- 分类删除时先统计 `document` 表中 `category_id = 当前分类ID` 的文档数量，如果大于 0，则禁止删除并提示“该分类下存在文档，不能直接删除”。
- 标签新增时会校验名称不能为空、名称不能重复，颜色默认值为 `#62c7bd`。
- 标签编辑时允许修改名称和颜色，同样会校验重复名称。
- 标签删除时先删除 `document_tag_rel` 中对应关联，再删除 `document_tag` 中的标签记录，符合“删除标签时解除文档与标签关联”的需求。
- `HomeController.prepareManagePage(...)` 负责组装管理页面通用数据，减少分类和标签 Controller 中重复准备页面模型。
- `LookupView` 增加 `id` 字段，页面可以根据 ID 拼接编辑和删除地址。

验证结果：

- 执行 `mvn package` 构建成功。
- 管理员登录后访问 `/categories` 返回 HTTP 200。
- 管理员登录后访问 `/tags` 返回 HTTP 200。
- 分类新增、编辑、删除链路验证通过。
- 标签新增、编辑、删除链路验证通过。
- 临时测试数据已删除，没有留在数据库中。

遇到的问题：

- 页面早期分类和标签使用同一段静态表格，缺少 ID，无法完成真实编辑和删除。
- 删除分类和删除标签的业务规则不同，不能用完全相同的删除逻辑。

解决办法：

- 为管理列表的视图对象增加 ID，并在页面行操作中拼接编辑、删除地址。
- 分类删除前检查文档引用，保护已有文档数据。
- 标签删除前解除中间表关联，避免外键约束阻止删除。

答辩可讲点：

- 分类和标签是文档组织能力的基础，本阶段已经从静态原型进入真实数据库操作。
- 分类是一对多关系，删除时需要保护文档数据，因此有引用时禁止删除。
- 标签是多对多关系，删除标签时需要处理中间表，体现了关系建模和数据一致性考虑。
- 当前实现仍然保持服务端渲染，表单提交和重定向简单稳定，适合课程项目演示。

后续可优化：

- 阶段 5 将文档元数据接入数据库后，文档表单可以直接选择真实分类和多个标签。
- 可以为删除操作增加前端确认弹窗。
- 可以增加分页、按名称搜索分类和标签。

- 文档编辑时更新文档基础字段，并重建标签绑定关系。

- 普通用户只能访问 `upload_user_id` 属于自己的文档。

技术与方案：

- Controller 层使用 Spring MVC 表单提交和重定向。
- 数据访问使用 `DocumentService`、`DocumentTagRelService`、`DocumentTagService`、`UserService`。
- 文档标签绑定使用“先删除旧关联，再插入新关联”的方式，保证编辑后的标签关系与表单选择一致。
- 创建和更新文档时使用 `@Transactional`，避免文档保存成功但标签关联失败导致数据不一致。
- 标签绑定前会校验标签 ID 是否真实存在，避免外键约束异常。
- 查询列表时先从数据库读取文档，再根据当前用户角色、关键词、分类、标签进行过滤。
- 普通用户访问非本人文档时，详情、编辑、删除都会被拒绝并重定向回文档列表。

实现细节：

- 阶段 5 的“上传”是元数据创建，不保存真实文件。
- 新增文档时会保存标题、原始文件名、文件类型、分类、描述、上传人、上传时间、更新时间。
- `stored_filename` 和 `storage_path` 暂时生成占位值，阶段 6 接入真实文件上传后替换。
- 上传用户 ID 通过当前登录用户 username 查询 `sys_user` 表获得。
- 文档列表中的标签名通过 `document_tag_rel` 查询关联，再通过 `document_tag` 读取标签名称。
- 文档分类名通过 `document.category_id` 查询 `document_category`。
- 文档逻辑删除依赖 `document.deleted` 和 MyBatis-Plus `@TableLogic`。

- 文档元数据新增成功。
- 文档详情展示成功。

遇到的问题：

- 初版创建文档未加事务，如果标签关联失败，可能留下半成品文档记录。

解决办法：

- 保存标签关联前先校验标签 ID 是否存在。
- 为文档新增、编辑、删除方法增加 `@Transactional`。
- 临时测试数据已通过页面删除接口清理。

答辩可讲点：

- 文档模块已经从静态页面进入真实数据库管理阶段。

## 技术选型说明

### 为什么使用 Spring Boot 单体架构

- 课程实训项目更关注 CRUD、页面流转、权限控制和文件管理闭环。
- 单体架构部署和调试成本低。
- 团队成员可以按照模块分工开发 Controller、Service、Mapper 和页面。
- 相比完整前后端分离，Thymeleaf 服务端渲染减少联调复杂度。

答辩表达：

> 项目定位是课程实训，不追求复杂微服务或完整前后端分离。我们选择 Spring Boot 单体架构，是为了把重点放在文档管理核心业务、权限控制和文件处理上，同时保证项目可运行、可维护、可展示。

### 为什么使用 Session 而不是 Token 或 Spring Security

- 项目只需要基础登录认证和角色区分。
- Session 与服务端渲染页面配合简单。
- HandlerInterceptor 可以清晰实现未登录拦截和基础权限判断。
- 避免引入 Spring Security 复杂配置，降低学习和实现成本。

答辩表达：

> 我们的权限需求主要是区分管理员和普通用户，不涉及复杂 RBAC，因此采用 Session 加拦截器实现登录控制，既能满足需求，也符合课程项目复杂度。

### 为什么先使用本地文件系统

- 项目目标是本地文档管理系统。
- 本地存储实现简单，便于演示上传和下载。
- 数据库存储元数据，文件系统保存真实文件，职责清晰。
- 避免引入对象存储、分布式文件系统等额外复杂度。

答辩表达：

> 系统将文件实体保存在服务器本地磁盘，数据库只保存路径和元数据。这样既能保证文件上传下载功能真实可用，又能保持数据库结构清晰。

### 为什么不做 OCR、Elasticsearch 和异步任务

- OCR 依赖外部工具和复杂文档处理流程。
- Elasticsearch 超出基础 CRUD 项目范围。
- 异步任务通常需要 Redis 或任务队列，增加部署和调试成本。
- 本项目更关注文档上传、分类、标签、搜索、权限和下载的基础闭环。

答辩表达：

> paperless-ngx 是成熟复杂的文档管理系统，但课程实训需要合理裁剪。我们保留文档管理核心思想，去掉 OCR、Elasticsearch、异步任务等复杂能力，保证项目范围可控、功能完整可演示。
