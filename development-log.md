# Development Log

> 本文件用于记录开发过程，重点服务于项目后期答辩、过程复盘和成果展示。它不同于 `progress.md`：`progress.md` 关注任务状态，本文件关注“我们做了什么、为什么这样做、遇到了什么问题、如何解决、答辩时如何讲清楚”。

## 使用方式

每完成一个阶段或重要功能后，建议补充以下内容：

- 本阶段目标。
- 实际完成内容。
- 采用的技术方案。
- 为什么选择该方案。
- 遇到的问题。
- 解决办法。
- 可用于答辩展示的亮点。
- 尚未完成或后续可优化的地方。

建议每次开发结束后追加一条“阶段记录”或“功能记录”，不要只在项目结束时补写。这样答辩时能更自然地说明开发过程。

## 项目基本信息

- 项目名称：本地文档管理系统。
- 参考项目：paperless-ngx。
- 项目类型：Java Web 课程实训项目。
- 开发策略：先落地基础可访问界面，再逐步完善真实业务功能。
- 主要技术：Spring Boot 3、Spring MVC、Thymeleaf、Session、HandlerInterceptor、MyBatis-Plus、MySQL、本地文件系统。
- 核心目标：实现文档上传、分类、标签、查询、下载、权限控制、统计和基础日志。

## 开发策略说明

本项目采用“先落地，再完善”的开发策略。

原因：

- 课程实训项目需要尽早形成可运行、可演示的系统。
- 先做界面可以帮助团队统一页面结构、菜单、路由和交互流程。
- 页面骨架完成后，各成员可以围绕明确入口并行开发后端功能。
- 前期使用模拟数据能减少数据库和业务逻辑未完成带来的阻塞。
- 后期再逐步替换为真实数据库、真实文件上传下载和权限控制，风险更可控。

答辩表达：

> 我们没有一开始就直接堆复杂功能，而是先完成可访问的系统原型，确定整体页面和业务流程，再逐步把模拟数据替换为真实数据，把页面操作接入数据库、文件系统和权限控制。这样能保证项目每个阶段都有可运行成果，也方便团队协作和阶段性验收。

### 阶段 0：项目骨架与运行入口

日期：2026-06-26

阶段目标：

- 创建可以启动的 Spring Boot Web 项目。
- 提供一个基础首页访问入口。
- 确认 Maven 构建和页面访问正常。

完成内容：

- 创建 `pom.xml`，引入 Spring Boot Web、Thymeleaf、Validation、Lombok、Hutool 和测试依赖。
- 创建 `.gitignore`，排除 `target/`、日志、IDE 文件和上传目录。
- 创建项目启动类 `PaperlessLocalApplication`。
- 创建 `HomeController`，支持 `/` 和 `/dashboard` 访问首页。
- 创建 `application.yml`，配置端口、Thymeleaf、上传大小和应用上传目录。
- 创建 `dashboard.html`，作为阶段 0 的基础首页。
- 创建 `app.css`，提供后台管理系统的基础布局和样式。

技术与方案：

- 使用 Spring Boot 3 作为后端基础框架。
- 使用 Thymeleaf 渲染首页，符合项目服务端渲染的技术路线。
- 阶段 0 暂不引入 MyBatis-Plus 和 MySQL 依赖，避免未配置数据源导致应用无法启动。
- 页面先展示基础统计占位数据，为阶段 1 的界面落地和后续真实统计预留结构。

方案理由：

- 阶段 0 的核心是“能启动、能访问”，不应被数据库配置阻塞。
- 首页先采用简单后台布局，后续登录页、文档列表、分类标签页面可以沿用同一视觉方向。
- 将数据库相关依赖推迟到阶段 3，更符合“先落地、再完善”的开发方案。

验证结果：

- 执行 `mvn package` 构建成功。
- 临时启动 `target/paperless-local-0.0.1-SNAPSHOT.jar` 后访问 `http://localhost:8080/dashboard`，返回 HTTP 200。
- 页面内容包含“阶段 0 已完成”，说明 Thymeleaf 模板渲染正常。

遇到的问题：

- 如果过早加入 MyBatis-Plus 和 MySQL 依赖，Spring Boot 可能会在没有数据源配置时启动失败。

解决办法：

- 阶段 0 仅保留 Web、模板、校验和工具类依赖。
- 将 MyBatis-Plus、MySQL 数据源配置放到阶段 3 接入数据库时处理。

答辩可讲点：

- 项目从第一步就保证可运行、可访问，而不是只停留在代码结构。
- 技术依赖按阶段引入，避免未完成模块影响当前系统启动。
- 首页虽然是基础版本，但已经体现后台管理系统的布局方向，为后续模块页面统一奠定基础。

后续可优化：

- 阶段 1 增加登录页、文档列表页、上传页、分类页、标签页等基础界面。
- 抽取公共布局模板，减少页面重复代码。
- 使用模拟数据完善首页统计和最近文档列表。

## 技术选型说明

### 为什么使用 Spring Boot 单体架构

- 课程实训项目更关注 CRUD、页面流转、权限控制和文件管理闭环。
- 单体架构部署和调试成本低。
- 团队成员可以按照模块分工开发 Controller、Service、Mapper 和页面。
- 相比完整前后端分离，Thymeleaf 服务端渲染减少联调复杂度。

答辩表达：

> 项目定位是课程实训，不追求复杂微服务或完整前后端分离。我们选择 Spring Boot 单体架构，是为了把重点放在文档管理核心业务、权限控制和文件处理上，同时保证项目可运行、可维护、可展示。

### 为什么使用 Session 而不是 Token 或 Spring Security

- 项目只需要基础登录认证和角色区分。
- Session 与服务端渲染页面配合简单。
- HandlerInterceptor 可以清晰实现未登录拦截和基础权限判断。
- 避免引入 Spring Security 复杂配置，降低学习和实现成本。

答辩表达：

> 我们的权限需求主要是区分管理员和普通用户，不涉及复杂 RBAC，因此采用 Session 加拦截器实现登录控制，既能满足需求，也符合课程项目复杂度。

### 为什么先使用本地文件系统

- 项目目标是本地文档管理系统。
- 本地存储实现简单，便于演示上传和下载。
- 数据库存储元数据，文件系统保存真实文件，职责清晰。
- 避免引入对象存储、分布式文件系统等额外复杂度。

答辩表达：

> 系统将文件实体保存在服务器本地磁盘，数据库只保存路径和元数据。这样既能保证文件上传下载功能真实可用，又能保持数据库结构清晰。

### 为什么不做 OCR、Elasticsearch 和异步任务

- OCR 依赖外部工具和复杂文档处理流程。
- Elasticsearch 超出基础 CRUD 项目范围。
- 异步任务通常需要 Redis 或任务队列，增加部署和调试成本。
- 本项目更关注文档上传、分类、标签、搜索、权限和下载的基础闭环。

答辩表达：

> paperless-ngx 是成熟复杂的文档管理系统，但课程实训需要合理裁剪。我们保留文档管理核心思想，去掉 OCR、Elasticsearch、异步任务等复杂能力，保证项目范围可控、功能完整可演示。
